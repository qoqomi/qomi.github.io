## 📌 이미지 프리로딩과 브라우저 캐시 동작 이해하기

### 1. 문제 상황

이미지 로딩 최적화를 진행하면서 `new Image()`를 활용한 프리로딩 방식을 적용하려고 했으나, 내부 동작에 대한 이해가 부족한 상태였다.

특히 아래와 같은 표현이 직관적으로 이해되지 않았다.

- “렌더링되지 않는다”
- “다운로드만 한다”

초기에는  
“DOM에 붙지 않으면 아무 일도 일어나지 않는 것 아닌가?”  
라는 오해가 있었고, 실제로 프리로딩이 어떤 효과를 만드는지 확신하기 어려웠다.

이로 인해 기능 적용 여부를 판단하기 위해 브라우저 동작 흐름부터 다시 확인하게 되었다.

---

### 2. 원인 분석

문제의 핵심은 이미지 로딩 과정을 하나의 단계로 인식하고 있었다는 점이었다.

실제로 브라우저에서 이미지가 표시되는 과정은 다음과 같이 분리되어 있다.

1. DOM에 `<img>` 요소가 추가됨  
2. 이미지 URL을 기반으로 네트워크 요청 발생  
3. 이미지 파일 다운로드  
4. 화면에 렌더링  

이 중에서

- 다운로드 (Network Layer)
- 렌더링 (Rendering Layer)

은 완전히 다른 단계였다.

하지만 기존에는 이 두 과정을 하나로 이해하고 있었기 때문에  
“렌더링되지 않으면 다운로드도 안 된다”는 잘못된 가정을 하고 있었다.

---

### 3. 해결 과정

#### 3-1. `new Image()` 동작 검증

아래 코드를 기준으로 실제 동작을 확인했다.

```js
const img = new Image();
img.src = url;
```
이 코드는 DOM에 추가되지 않지만, src를 설정하는 순간 브라우저는 다음 동작을 수행한다.

네트워크 요청 발생
이미지 다운로드
캐시에 저장

즉,

```
렌더링 ❌
다운로드 ✔️
```
라는 상태가 성립한다는 것을 확인했다.

#### 3-2. 브라우저 캐시 재사용 확인

다음 단계에서는 캐시가 실제로 활용되는지 검증했다.
```
// 프리로딩
new Image().src = "a.jpg";
```
```
// 이후 렌더링
<img src="a.jpg" />
```
Network 탭을 통해 확인한 결과:

1. 첫 요청: 이미지 다운로드 발생
2. 두 번째 요청: (from disk cache)로 표시

이를 통해 동일한 URL 요청 시 네트워크 요청이 발생하지 않고 캐시가 재사용된다는 것을 확인했다.

#### 3-3. 실제 UI 시나리오 적용

카드 리스트 구조에 적용해 검증했다.

API로 이미지 5개 수신
초기 렌더링: 2개
이후 버튼 클릭 시 3개 추가 노출
**(1) 프리로딩 미적용**
버튼 클릭 시점에 3~5번 이미지 요청 발생
평균 300~500ms 지연 발생 (네트워크 환경 기준)

→ 사용자 체감: 카드가 늦게 나타남

**(2) 프리로딩 적용**
posts.slice(2).forEach((p) => preloadImage(p.url));
초기 시점에 3~5번 이미지 미리 다운로드
버튼 클릭 시 네트워크 요청 없음

→ 결과: 렌더링 지연 거의 0ms 수준으로 감소
→ 사용자 체감상 즉시 표시

### 4. 결과

프리로딩 적용 이후 다음과 같은 개선을 확인했다.

추가 카드 노출 시 네트워크 요청 제거 (100% 감소)
클릭 후 이미지 표시까지의 지연 시간
→ 약 300~500ms → 0~50ms 수준으로 감소
사용자 체감 속도 개선 (지연 없이 즉시 표시)

결과적으로 프리로딩은 단순한 최적화가 아니라
렌더링 시점의 비용을 사전에 분산시키는 전략임을 확인했다.

### 5. 한계 및 시행착오
#### 5-1. URL 기반 캐시 한계

다음과 같은 경우 캐시가 동작하지 않았다.
```
a.jpg?t=123
a.jpg?t=456
```

- 동일한 이미지라도 URL이 다르면 다른 리소스로 인식
프리로딩 효과 완전히 무효화

- 초기 테스트에서 캐시가 적용되지 않아 원인을 추적했는데,
쿼리 파라미터가 매번 달라지는 구조 때문이었다.

#### 5-2. 불필요한 네트워크 사용 가능성

프리로딩은 실제로 사용하지 않을 이미지까지 다운로드할 수 있다는 문제가 있다.

사용자가 버튼을 클릭하지 않는 경우 → 불필요한 트래픽 발생

따라서

사용자 행동이 예측 가능한 경우
노출 확률이 높은 리소스

에 한정하여 적용하는 것이 필요했다.

#### 5-3. “렌더링되지 않는다”에 대한 오해

초기에는

DOM에 없으면 아무 일도 안 일어난다

고 생각했지만,

실제로는

리소스 로딩은 DOM과 독립적으로 동작한다

는 점을 놓치고 있었다.

이 부분이 개념적으로 가장 큰 시행착오였다.

### 6. 정리 및 배운 점

이번 경험을 통해 다음과 같은 점을 명확히 이해하게 되었다.

- 이미지 로딩은 다운로드와 렌더링이 분리된 과정이다
new Image()는 렌더링 없이 네트워크 요청만 수행한다
- 다운로드된 리소스는 브라우저 캐시에 저장된다
- 동일한 URL 요청 시 네트워크 없이 즉시 렌더링 가능하다

### 💡 결론

이미지 프리로딩은
브라우저 캐시를 활용해 렌더링 시점의 네트워크 비용을 제거하고,
사용자 경험을 개선하는 전략이다.

다만,

- URL 일관성
- 리소스 사용 가능성

을 고려하지 않으면 오히려 비효율이 발생할 수 있어
선별적 적용이 필요하다는 한계도 함께 확인했다.

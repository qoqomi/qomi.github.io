## 1. 문제 상황

이미지 로딩 최적화를 진행하면서 `new Image()`를 활용한 프리로딩 방식을 적용하려고 했으나, "렌더링되지 않는다", "다운로드만 한다"는 표현이 직관적으로 이해되지 않았다. DOM에 붙지 않으면 아무 일도 일어나지 않는다는 오해가 있었고, 브라우저 동작 흐름부터 다시 확인하게 되었다.

---

## 2. 원인 분석

브라우저에서 이미지가 표시되는 과정은 다음과 같이 분리되어 있다.

1. DOM에 `<img>` 요소 추가
2. 이미지 URL 기반 네트워크 요청 발생
3. 이미지 파일 다운로드
4. 화면에 렌더링

다운로드(Network Layer)와 렌더링(Rendering Layer)은 완전히 다른 단계다. "렌더링되지 않으면 다운로드도 안 된다"는 가정이 잘못된 것이었다.

---

## 3. 해결 과정

### 3-1. `new Image()` 동작 검증

```js
const img = new Image();
img.src = url;
```

DOM에 추가되지 않아도 `src`를 설정하는 순간 브라우저는 네트워크 요청을 발생시키고 이미지를 다운로드해 캐시에 저장한다. 즉, 렌더링 없이 다운로드만 일어난다.

### 3-2. 브라우저 캐시 재사용 확인

```js
// 프리로딩
new Image().src = "a.jpg";

// 이후 렌더링
<img src="a.jpg" />
```

Network 탭에서 첫 요청은 다운로드가 발생하고, 두 번째 요청은 `(from disk cache)`로 표시됐다. 동일 URL이면 네트워크 요청 없이 캐시가 재사용된다.

### 3-3. 실제 UI 시나리오 적용

API로 이미지 5개를 수신하고 초기에 2개만 렌더링, 버튼 클릭 시 3개를 추가 노출하는 구조에서 비교했다.

**프리로딩 미적용**: 버튼 클릭 시점에 3~5번 이미지 요청 발생, 평균 300~500ms 지연.

**프리로딩 적용**: 초기 시점에 `posts.slice(2).forEach(p => preloadImage(p.url))`로 미리 다운로드. 버튼 클릭 시 네트워크 요청 없이 0~50ms 수준으로 즉시 표시.

---

## 4. 결과

- 추가 카드 노출 시 네트워크 요청 제거 (100% 감소)
- 이미지 표시 지연 300~500ms → 0~50ms로 감소
- 프리로딩은 단순 최적화가 아니라 렌더링 시점의 비용을 사전에 분산시키는 전략이다.

---

## 5. 한계 및 시행착오

**URL 기반 캐시 한계**: `a.jpg?t=123`과 `a.jpg?t=456`처럼 쿼리 파라미터가 달라지면 다른 리소스로 인식해 캐시가 무효화된다.

**불필요한 네트워크 사용**: 사용자가 버튼을 클릭하지 않으면 미리 다운로드한 이미지가 불필요한 트래픽이 된다. 사용자 행동이 예측 가능하고 노출 확률이 높은 리소스에만 선별 적용이 필요하다.

---

## 6. 결론

이미지 프리로딩은 브라우저 캐시를 활용해 렌더링 시점의 네트워크 비용을 제거하고 사용자 경험을 개선하는 전략이다. 단, URL 일관성과 리소스 사용 가능성을 고려하지 않으면 오히려 비효율이 생길 수 있다.
